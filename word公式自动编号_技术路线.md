# Word 行间公式快捷键插入 + 自动章节编号 + 正文引用——技术路线

## 1. 需求边界

- 快捷键/面板按钮触发,插入一个新的公式行
- 公式编辑使用 Word **自带的公式编辑器**(原生 OMath 对象),不做 LaTeX 解析/转换
- 编号自动按用户选择的格式生成,位于公式行最右侧,**最多支持二级(章节-流水号)**,不支持三级编号
- 编号底层用 Word 原生**题注(Caption)系统**,而不是插件自己计算编号文本
- 提供面板按钮,在正文中插入"指向某个公式编号"的交叉引用,引用随编号变化自动联动

不做的事:不写 LaTeX 引擎、不支持三级及以上编号格式、不自己维护一套独立于 Word 域系统的编号计算引擎。

## 2. 关键决策:为什么是题注,不是插件自算编号

最初考虑过"插件自己扫描文档、计算编号、写入纯文本"的方案,但存在一个致命问题:**正文里引用编号的地方不会跟着联动**。例如用户在前面插入一段新公式,后面所有公式编号顺延变化,但正文中"如公式 2-3 所示"这类引用如果是纯文本,不会自动更新,除非额外实现一套引用追踪系统,工作量陡增。

Word 的**题注 + 交叉引用**是配套设计的:

- 题注生成编号(基于域代码)
- 正文交叉引用指向同一个编号源(也是域代码)
- 刷新域时,编号和所有引用点**自动同步**

因此放弃自算编号,改为复用 Word 原生题注系统,引用功能直接获得免费的联动能力。

## 3. 题注编号格式的能力边界

Word 题注原生支持的格式:

| 格式 | 示例 | 题注原生支持 |
|---|---|---|
| 纯流水号(不分章节) | `1, 2, 3` | ✅ 不勾选"包含章节号" |
| 章节-流水号(二级) | `2-3` / `2.3` | ✅ 勾选"包含章节号",分隔符可选(连字符/句点/冒号等) |
| 三级编号 | `1.1.3` | ❌ 不支持,题注只能取一级标题编号 + 流水号两段 |

**本期方案明确放弃三级编号**,只支持上述前两种格式。这个决策大幅降低了实现复杂度,使方案可以完全建立在 Word 原生题注/域机制之上,不需要手动拼接多重 `STYLEREF` 域。

## 4. 前提条件

题注"包含章节号"功能依赖标题样式已关联多级列表编号:

- 文档章节标题使用 Word 内置「标题 1」(或用户指定的某一级标题)样式
- 该标题样式已关联「多级列表」编号方案(显示为自动编号的 1、2、3,而非手动打字)

面板/宏中应加入检测逻辑,若文档未满足前提,提示用户先设置标题样式,而不是静默失败。

## 5. 技术选型

| 模块 | 方案 | 理由 |
|---|---|---|
| 插件载体 | Office Add-in(Office.js + 任务窗格) | 需要图形面板做格式选择和引用插入按钮,VBA 的 UserForm 体验和维护成本不如网页技术栈 |
| 公式输入 | 插入原生 OMath 对象(必要时退化为直接插入 OOXML `<m:oMath>` 片段) | 复用 Word 自带公式编辑器;Office.js 对 OMath 的原生 API 支持有限,需提前验证可行性,这是本方案唯一的技术不确定点 |
| 行内布局 | 段落制表位(Tab Stop):居中 + 右对齐 | 公式居中、编号顶右,贴近 LaTeX 排版习惯 |
| 编号格式选择 | 面板提供单选:纯数字 / 章节-流水号(可选分隔符) | 对应题注原生支持的两种格式,不暴露三级选项 |
| 编号生成 | 调用题注插入逻辑,生成 `STYLEREF` + `SEQ` 域代码 | 编号联动、章节重置交给 Word 原生引擎,可靠性高 |
| 正文引用插入 | 面板"插入引用"按钮,调用交叉引用(`REF` 域),引用对应公式题注 | 与编号联动,无需自建追踪系统 |
| 编号/引用刷新 | `Fields.Update`,绑定快捷键或面板按钮 | 域更新非实时,需手动触发,这是 Word 域系统的固有特性 |

## 6. 核心实现逻辑

### 6.1 插入公式行(快捷键或面板按钮)

```
1. 光标处插入新段落
2. 设置该段落制表位:
   - 制表位 A:居中对齐,位置据当前页面宽度/边距动态计算
   - 制表位 B:右对齐,位置据当前页面右边距动态计算
3. 插入 Tab → 跳到制表位 A
4. 插入空白 OMath 对象,光标定位进入编辑态,等待用户输入公式
5. 在公式后插入 Tab → 跳到制表位 B
6. 按用户在面板选择的格式,插入题注(底层为 STYLEREF + SEQ 域),
   外层包一层括号渲染成 (2-3) 这种习惯格式
7. 光标移回公式内部
```

### 6.2 插入正文引用(面板按钮"插入公式引用")

```
1. 弹出列表,展示文档中已有的公式题注(按编号排序供用户选择)
2. 用户选定后,在光标处插入交叉引用(REF 域),指向该题注的书签
3. 引用文本随后续编号变化,在刷新域时自动同步
```

### 6.3 刷新编号与引用

```
调用 Fields.Update,遍历文档所有域(题注编号 + 交叉引用)统一刷新
```

## 7. 关键风险点

| 风险 | 说明 | 缓解方式 |
|---|---|---|
| 标题未应用编号样式 | `STYLEREF` 取不到章节号,题注报错或异常 | 面板内做前置检测,给出修复提示 |
| 编号/引用不实时 | 插入/删除公式后不会自动跳变 | 提供刷新按钮,用户养成习惯;可考虑在文档保存前自动触发一次 Update |
| 制表位位置写死 | 不同页边距/纸张下,居中/右对齐位置可能不准 | 动态读取页面宽度和边距实时计算制表位坐标,不硬编码 |
| Office.js 操作 OMath 的 API 局限 | 可能无法直接调用原生公式插入 API | 需提前做技术验证,必要时改为直接插入 OOXML 片段 |
| 引用列表随公式增多变长 | 面板里选择引用目标的列表体验变差 | 可加搜索/筛选,或按章节分组展示 |

## 8. 工作量评估

- Office.js 插入原生 OMath 对象的可行性验证:**0.5-1 天**(优先做,决定后续方案是否需要调整)
- 面板 UI(格式选择 + 插入引用列表):**1.5-2 天**
- 公式行插入逻辑(制表位 + 公式 + 题注):**1-2 天**
- 正文引用插入逻辑:**1 天**
- 前置条件检测、动态制表位计算、边界情况调试:**2-3 天**

整体落地约 **1-1.5 周**,相比"插件自算编号"的方案明显更轻,核心原因是把编号联动和引用联动的复杂度都交给了 Word 原生域系统。

## 9. 后续可扩展方向(本期不做)

- 跨平台支持(Mac/Web 版 Word):需验证 Office.js 在不同平台上对 OMath 和域操作的兼容性差异
- LaTeX 输入支持:引入 LaTeX → OMML 转换链路(LaTeX → MathML → OMML,可参考微软 `MML2OMML.XSL`)
- 三级编号支持:如未来确有需求,需手动拼接多重 `STYLEREF` 域,绕开题注图形界面直接操作域代码字符串
